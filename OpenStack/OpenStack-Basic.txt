OpenStack
=========
OpenStack is an open-source cloud platform used to create and manage Infrastructure as a Service (IaaS).
  It provides:
	- Virtual Machines
	- Networking
	- Storage
	- Identity management
	- Image management
    Similar to how AWS provides EC2, EBS, VPC, etc.
    
    Using OpenStack, users can:
	- launch VMs
	- create networks
	- attach storage
	- manage users/projects
	- automate infrastructure

TripleO in OpenStack
	TripleO stands for: OpenStack-On-OpenStack
	

Undercloud (director)
	- Small OpenStack environment
	- Used for provisioning and deployment

Overcloud	
	- Actual production OpenStack cloud
	- Production OpenStack cluster
	- Controller + Compute + Storage nodes

Storage
Nodes that provide storage for the OpenStack environment. This includes nodes for:
	Ceph Storage nodes:
		- Used to form storage clusters. Each node contains a Ceph Object Storage Daemon (OSD). In addition, the director installs Ceph Monitor onto the Controller nodes in situations where it deploys Ceph Storage nodes.
		- Block storage (cinder) - Used as external block storage for HA Controller nodes. This node contains the following components:

			OpenStack Block Storage (cinder) volume
			OpenStack Telemetry (ceilometer) agent
			Open vSwitch.
	
	Object storage (swift) - These nodes provide a external storage layer for Openstack Swift. The Controller nodes access these nodes through the Swift proxy. This node contains the following components:
			OpenStack Object Storage (swift) storage
			OpenStack Telemetry (ceilometer) agent
			Open vSwitch.
			

High Availability 
The Red Hat OpenStack Platform director uses a Controller node cluster to provide high availability services to your OpenStack Platform environment. 
The director installs a duplicate set of components on each Controller node and manages them together as a single service. 
This type of cluster configuration provides a fallback in the event of operational failures on a single Controller node; 
this provides OpenStack users with a certain degree of continuous operation.

The OpenStack Platform director uses some key pieces of software to manage components on the Controller node:
	Pacemaker - Pacemaker is a cluster resource manager. Pacemaker manages and monitors the availability of OpenStack components across all nodes in the cluster.
	HAProxy - Provides load balancing and proxy services to the cluster.
	Galera - Replicates the Red Hat OpenStack Platform database across the cluster.
	Memcached - Provides database caching.

TroubleShooting Questions
	What Happens if Nova Compute Service Goes Down?
		Instances already running may continue working, but new VM operations on that compute node will fail until the service is restored.

	What is Live Migration?
		Live migration moves a running VM from one compute node to another with minimal downtime.
	
	What is Evacuation?
		Evacuation recovers VMs from a failed compute node by restarting them on another available compute node.

OpenStack Troubleshooting Interview Questions
=============================================
Q) A VM is stuck in BUILD state. How would you troubleshoot?

First, I would check the VM status and fault details using the OpenStack CLI.

Then I would verify whether the Nova compute service is up and whether the scheduler successfully selected a compute node.

After that, I would check Nova logs, resource availability on compute nodes, image accessibility from Glance, and networking allocation from Neutron.

I would also verify RabbitMQ communication and confirm there are no storage-related issues if the VM is booting from volume.


Important Checks
Commands:
	openstack server show <vm>
	openstack compute service list
	openstack hypervisor list

Logs:
	/var/log/nova/nova-compute.log
	/var/log/nova/nova-scheduler.log


Q) VM Not Reachable Through Network
	or Instance is ACTIVE but not reachable. What will you check?

I would first verify whether the instance has the correct IP address and whether security groups allow required traffic such as SSH or ICMP.

Then I would check floating IP association, router connectivity, Neutron agent status, and DHCP functionality.

I would also validate Open vSwitch or Linux bridge configuration and verify routing and namespace connectivity on the network node.

Important Checks

Commands:
openstack server show <vm>
openstack security group rule list
openstack floating ip list
openstack network agent list

Linux:
ip netns
ovs-vsctl show
ip a

Q) Nova Compute Service Down
	or Nova compute service is down. What would you do?

I would first verify the service status using systemctl and OpenStack service listing commands.

Then I would check logs for errors related to libvirt, RabbitMQ, resource exhaustion, or configuration problems.

I would also verify hypervisor connectivity and ensure the compute node has sufficient CPU, memory, and storage resources.

Commands
systemctl status openstack-nova-compute
openstack compute service list


Q) Volume Attach Failure
Question
	“Cinder volume is not attaching to VM. How would you troubleshoot?”

I would first check the volume state and verify whether the Cinder services are healthy.

Then I would review backend storage connectivity, validate iSCSI or Ceph backend health, and inspect compute node logs for attachment-related errors.

I would also verify multipath configuration and confirm there are no permission or network connectivity issues between compute and storage nodes.

Commands
	openstack volume list
	openstack volume show <volume>
	ceph -s


Q) Ceph Cluster HEALTH_WARN
Question

“Ceph cluster shows HEALTH_WARN. What will you check?”

Good Answer
I would first run ceph health detail to identify the exact warning.

Then I would check OSD status, degraded PGs, utilization imbalance, recovery operations, and monitor quorum status.

Depending on the issue, I would verify whether any OSDs are down, nearfull, or experiencing slow operations.
Commands
ceph -s
ceph health detail
ceph osd tree
ceph pg stat


Q) RabbitMQ Failure in OpenStack
Question

“How does RabbitMQ impact OpenStack?”

Answer
RabbitMQ is used for communication between OpenStack services. If RabbitMQ fails, inter-service communication may stop, causing VM operations and API workflows to fail.

Q) Compute Node Full
Question
	“Compute node storage is full. What will happen?”

Good Answer
New VM creation may fail, instance operations may become unstable, and services could experience performance degradation.

I would identify disk utilization sources such as logs, unused images, ephemeral disks, or snapshots, and perform cleanup after validation.


Q) Neutron Agent Down
Question
“Neutron L3 or DHCP agent is down. What impact can happen?”

Answer
Tenant routing, DHCP allocation, floating IP functionality, or external connectivity may fail depending on which Neutron agent is affected.

Q) VM Performance Slow
Question
	“A VM is very slow. How would you troubleshoot?”

I would check compute node resource utilization such as CPU, memory, and disk I/O.

Then I would verify storage backend performance, Ceph health, noisy-neighbor workloads, and hypervisor-level resource contention.

I would also review VM flavor allocation and inspect system logs for hardware or storage latency issues.
