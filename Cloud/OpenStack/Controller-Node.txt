Openstack patching
==================
In our environment, we use RHN for repository and patch management to pull certified Red Hat updates. We follow a rolling patching approach starting with the director, then controller nodes, and finally compute nodes. Before patching compute nodes, we disable the compute service, migrate workloads if required, apply updates, restart services, and validate each node before moving to the next. After completion, we perform overall OpenStack health checks to ensure all services and instances are stable.

Q) Instance UNKNOWN, SHUTOFF and ERROR stage?

“UNKNOWN means OpenStack cannot determine VM status, usually due to compute or communication failure. SHUTOFF means the VM is powered off intentionally or during maintenance. ERROR means the VM failed due to resource, storage, or hypervisor issues.”



VM Migration
============
In OpenStack, VM migration means moving an instance from one compute host to another.
There are mainly three types of migrations:
1. Cold Migration
	The VM is powered OFF before migration.
	
	Process
	Shut down VM
	Copy/move instance
	Start VM on another compute node

	Features
	Simple and reliable
	Minimal resource requirement
	Downtime involved

	Use Cases
	Hardware maintenance
	Resource balancing
	Failed compute recovery
	
	# openstack server migrate <vm-name>
	
2. Live Migration

The VM is moved while running with minimal downtime.
Process
	Memory pages copied continuously
	Final synchronization occurs
	VM switches to target host
Features
	Near-zero downtime
	No application interruption
	Requires shared storage or block migration
Requirements
	Shared storage (NFS/Ceph) OR block migration
	Same CPU compatibility
	SSH connectivity between compute nodes
Use Cases
	Maintenance without downtime
	Load balancing
	Scaling operations
	
	# openstack server migrate --live <target-host> <vm-name>

3. Block Migration
	A type of live migration where local disk data is also copied.

Features
	Used when shared storage is NOT available
	Copies VM disk + memory
	Slower than normal live migration
Use Cases
	Local storage environments
	Small deployments
	
	# openstack server migrate --live <target-host> --block-migration <vm-name>
	
Migration Comparison
====================
Type			VM State	Downtime	Shared Storage Needed
Cold Migration	Powered 	Off	Yes		No
Live Migration	Running		Very Low	Usually Yes
Block Migration	Running		Very Low	No
