Q)Pod to pod communication
	Pods on the SAME Node
		Flow : Pod A → virtual network → Pod B

	- Pods communicate using IP addresses, but Kubernetes uses a virtual network internally to connect those pod IPs and route 	traffic between pods.
	- Pods can communicate using those IPs.
	- No NAT is required between pods.

	CNI plugins (Container Network Interface)
	Examples:
		Flannel
		Calico
		Weave

Q)Pod Communication on DIFFERENT Nodes
		Pods communicate using pod IPs, and the CNI plugin handles the networking and routing between pods across the Kubernetes cluster.”

Q)How to troubleshoot node if it stuck somewhere. and you did not found any error.
	First, I would verify whether the node is reachable through SSH or ping. Then I would check system resources like CPU, memory, disk, and load average to identify possible resource exhaustion.

	or

	I would first check node reachability, resource utilization, logs, kubelet/container runtime status, and node conditions. If no error is visible, I would compare the node with healthy nodes and isolate whether the issue is related to resources, networking, kubelet, or the operating system.”


Q)How to troubleshoot deployment. what is your approach.
	1. Check Deployment Status
		Available replicas
		Ready pods

	2. Describe Deployment
		checks:
			Events
			Image issues
			Scheduling issues
			Replica errors
	3. Check Pods
		check pods
			CrashLoopBackOff
			Pending
			Error
			ImagePullBackOff

	4. Describe Pod
		Check:
			Events section
			Resource issues
			Node scheduling problems
			Volume errors

	5. Check Pod Logs
		Check:
			Application crash
			Config issue
			DB connection failure
			Port issue

	To troubleshoot a deployment, I check deployment status, describe the deployment and pods, review pod logs and events, verify services and resource availability, and identify whether the issue is related to image, configuration, application, or node health.

Q) Post-check before openShift-upgrade and process.
	- etcd backup
	- oc get co, nodes, mcp verify

	1. Check Cluster Version
		Verify:
			Current version
			Available updates
			Cluster status
	2. Check All Nodes
		oc get nodes
		Ensure:
			All nodes are Ready
			No node in NotReady

	3. Check MCP Status
		oc get mcp
		Important:
			UPDATED=True
			UPDATING=False
			DEGRADED=False
		
		Cluster should not already be updating/degraded.

	4. Check Cluster Operators
		oc get co
			All operators should be:
				AVAILABLE=True
				DEGRADED=False
				PROGRESSING=False

	5. Check Pending Pods / Failed Pods
		oc get pods -A
		Check for:
			CrashLoopBackOff
			Pending
			Error states

	6. Backup Important Resources
		Usually:
			ETCD backup
			Important configs
			YAML exports
		ETCD backup:
			cluster-backup.sh

	Before OpenShift upgrade, I verify cluster health, nodes, MCP, cluster operators, pod status, alerts, and take backups. Then I start the upgrade using oc adm upgrade and continuously monitor cluster operators and MCP until the upgrade completes successfully.

	For kubernetes
	
	Before Kubernetes upgrade, I verify node health, pod status, control plane components, resource usage, and take ETCD backup. Then I upgrade control plane and worker nodes one by one using drain and uncordon strategy while continuously monitoring cluster health.
