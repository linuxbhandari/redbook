1. EmptyDir Volume (Temporary Storage)
======================================
		-	Temporary storage created when Pod starts.
		- Data exists only during pod life and will deleted when Pod is removed

cat >  emptydir.yaml
---
apiVersion: v1
kind: Pod
metadata:
  name: emptydir-demo
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - mountPath: /data
      name: temp-storage

  volumes:
  - name: temp-storage
    emptyDir: {}
---

	How it works
		- Volume created when pod starts.
		- Deleted when pod is removed.

	👉 Use cases:
			- Cache
			- Temporary files
			- Shared data between containers

2. HostPath Volume (Node Storage)
=================================
		- A HostPath volume mounts a directory from the worker node filesystem into a Pod. So the pod directly uses the node’s disk.
		- Mostly for labs/debugging


	Important Warning (Very Important)
		- Works only on single node or testing environments
		- NOT recommended for production
		
		Use cases:
			- log collection
			- node monitoring
			- Docker socket access

--------
apiVersion: v1
kind: Pod
metadata:
  name: hostpath-demo
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: node-storage
      mountPath: /usr/share/nginx/html

  volumes:
  - name: node-storage
    hostPath:
      path: /data/webcontent
      type: DirectoryOrCreate
--------

3. PersistentVolume (PV)
========================
		- A PV is a cluster-wide storage resource used to store data permanently, even if Pods are deleted or restarted.
		- provide Independent of Pod lifecycle
		- Created by admin or dynamic provisioning

		PV Workflow : Pod → PVC → PV → Physical Storage
			- Pod requests storage using PVC
			- PVC (PersistentVolumeClaim) binds to PV
			- PV connects to actual storage

		Example Storage Backends
			- AWS EBS
			- Azure Disk
			- GCP Persistent Disk
			- NFS
			- Ceph
			- Local Disk

		Types of Provisioning

		1. Static Provisioning
				Admin manually creates PV
	
		2. Dynamic Provisioning
				Kubernetes automatically creates PV using StorageClass

Example:

Architecture
============
Pod
 ↓
PVC
 ↓
StorageClass
 ↓
NFS Provisioner
 ↓
NFS Server Storage


Step 1: Install NFS Server

Step 2: Add Helm Repo
	helm repo add nfs-subdir-external-provisioner https://kubernetes-sigs.github.io/nfs-subdir-external-provisioner/

Step 3: Install Provisioner
	helm install nfs-storage nfs-subdir-external-provisioner/nfs-subdir-external-provisioner --set nfs.server=kube-master --set nfs.path=/srv/nfs/kubedata -n nfs-provisioner

Step 4: Verify StorageClass
	kubectl get storageclass


Step 5: Create PVC
----
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: test-pvc
spec:
  accessModes:
    - ReadWriteMany
  storageClassName: nfs-client
  resources:
    requests:
      storage: 1Gi
----

kubectl apply -f pvc.yaml

Step 6: Test With Pod

-------------


PersistentVolumeClaim (PVC)
===============================
		Request for storage by application.
			- Pods use PVC
			- PVC binds to PV

	Access Modes in PVC
	===================
		Access Mode						Meaning
		ReadWriteOnce (RWO)		One node can mount read-write
		ReadOnlyMany (ROX)		Many nodes read-only
		ReadWriteMany (RWX)		Many nodes read-write
		ReadWriteOncePod			Single Pod only

	Q)What happens if PVC is deleted?
		Depends on policy:
			Retain → data remains
			Delete → storage deleted
			Recycle → cleaned

-----

4. StorageClass
		It enables dynamic provisioning of Persistent Volumes (PVs).
		Instead of manually creating PVs, Kubernetes automatically creates them when a PVC requests storage.

		A StorageClass defines:
			- how storage should be provisioned
			- what type of storage to create
			- which storage backend to use

Without StorageClass:
=====================
Admin creates PV manually
        ↓
Application creates PVC
        ↓
PVC binds to matching PV



With StorageClass:
===================
Application creates PVC
        ↓
StorageClass provisions PV automatically
        ↓
PVC binds dynamically


What Happens Internally
=======================
Flow:

PVC Created
    ↓
StorageClass Selected
    ↓
CSI Driver Called
    ↓
Disk Created
    ↓
PV Created Automatically
    ↓
PVC Bound
    ↓
Pod Uses Volume



5. CSI stands for: Container Storage Interface
		- It is a standard interface between Kubernetes and storage systems.
		- CSI allows Kubernetes to work with any storage provider using a common API.


6. ConfigMap : Mount configuration files into containers.
		A ConfigMap stores:
			- application configuration
			- environment settings
			- config files
			- command-line arguments
		
		separately from container images.

		It helps make applications configurable without rebuilding Docker images.

		Why ConfigMap Exists
				Without ConfigMap:
					Configuration hardcoded inside container image


	Q) Can ConfigMap updates reflect automatically?
			Method								Auto Update
			Environment variable	No
			Mounted volume				Yes


7. Secret : A Secret stores sensitive information securely in Kubernetes.
			Examples:
				passwords
				API keys
				SSH keys
				TLS certificates
				tokens
				database credentials

		Why Secrets Exist
			Without Secrets:
					Passwords hardcoded in YAML or Docker image

		Problems:
			insecure
			exposed in Git
			difficult to rotate credentials
		
		Secrets solve this by separating sensitive data from application code.

		Kubernetes Secrets are: Base64 encoded. NOT encrypted by default.

		Types of Secrets
		Type																	Purpose
		Opaque																Generic secrets	
		kubernetes.io/tls											TLS certs
		kubernetes.io/dockerconfigjson				Container registry auth
		kubernetes.io/service-account-token		ServiceAccount token
		kubernetes.io/ssh-auth								SSH keys

		1. Generic Secret (Opaque)
			Create Secret from CLI
				# kubectl create secret generic db-secret --from-literal=username=admin --from-literal=password=mysecret

			Encode Values
				# echo -n 'mypassword' | base64

			commands
			kubectl get secret
			kubectl describe secret db-secret
			kubectl get secret db-secret -o yaml
			echo 'bXlwYXNzd29yZA==' | base64 -d
