Q) kubernetes 
	- Kubernetes (K8s) is an open-source container orchestration platform that helps manage containerized applications by automating deployment, scaling, load balancing, networking, and self-healing across a cluster of machines.

Q) What are the main components of Kubernetes architecture?
    Kubernetes architecture consists of two main parts:
    1. Control Plane Components : The control plane manages the overall state of the Kubernetes cluster.
		API Server
            - The entry point for all Kubernetes operations. 
            - Handles communication between all Kubernetes components.
            - Performs authentication and authorization.
            - Exposes the Kubernetes API to users and cluster components.
            - Validates API requests.
            - Validates and processes resource definitions (YAML/JSON manifests).
            - Persists cluster state information in etcd.

		Scheduler
            - Responsible for assigning Pods to worker nodes.
            - Monitors newly created Pods without an assigned node.
            - Selects the most suitable node based on resource availability and scheduling policies.
            - Considers factors such as CPU, memory, affinity/anti-affinity, taints, and tolerations.

		Controller Manager
            - Runs various Kubernetes controllers.
            - Continuously monitors the cluster's current state.
            - Compares the current state with the desired state.
            - Takes corrective actions to maintain the desired state.

		etcd
            - A distributed and highly available key-value store.
            - Stores all cluster configuration and state information.
            - Acts as the single source of truth for the Kubernetes cluster.
            - Used by the API Server to read and write cluster data.

	Worker Node Components : Worker nodes run the application workloads.

		Kubelet
            - Runs as an agent on every worker node and ensures Pods are running according to the pod specfications.
            - Registers the node with the API Server.
            - Monitors the health and status of the node and reports it to the API Server.
            - Communicates with the container runtime to manage containers. like (create, start, stop and delete containers)
            - Performs liveness, readiness, and startup probes on containers.
            - Restarts failed containers according to the Pod's restart policy.
            - Mounts and manages volumes for pods.
            - Ensures the actual state of Pods matches the desired state.
            - Collects resource usage information and node metrics.


		Kube Proxy
            - Manages network rules for Pod and Service communication.
            - Performs load balancing across backend Pods.
            - Routes traffic to the appropriate Pods.
            - Watches Service and Endpoint changes from the API Server.

            
		Container Runtime (containerd, CRI-O, etc.) 
            – Responsible for pulling container images from registries and running containers on the node.
            - Creates, starts, stops, and removes containers.
            - Manages the container lifecycle.


Q) What is a Pod?
    - A Pod is the smallest deployable unit in Kubernetes.
    - A Pod can contain one or more containers that run together.
        Containers within a Pod share:
            - Network namespace
            - Storage volumes
            - Same Ip address
            - Same hostname

	- Containers in the same Pod can communicate with each other using localhost.
    - Pods are ephemeral, meaning they can be created, deleted, or replaced at any time.

    Important Note:
    In production environments, Pods are usually not created directly. Instead, they are managed by controllers such as:
        - Deployment
        - StatefulSet
        - DaemonSet
        - Job 
        - CronJab

    These controllers automatically create, manage, scale, and recover Pods to maintain the desired state.

Q) What is the difference between a Pod and a Deployment?
    Pod
        - A Pod is the smallest deployable unit in Kubernetes.
        - Contains one or more containers.
        - Provides shared networking and storage for containers.
        - If a Pod is deleted, Kubernetes will not recreate it unless it is managed by a controller.
    Example:
        Pod
        └── Nginx Container

    Deployment
        - A Deployment is a Kubernetes controller that manages Pods.
        - Ensures the desired number of Pod replicas are running.
        - Automatically replaces failed Pods.
        - Supports rolling updates and rollbacks.
        - Used for deploying stateless applications.
    Example:
        Deployment
        ├── Pod-1
        ├── Pod-2
        └── Pod-3

    Interview One-Liner
        A Pod is the smallest deployable unit that runs one or more containers, whereas a Deployment is a controller that manages Pods by providing replica management, self-healing, scaling, rolling updates, and rollbacks.


Q) What is the difference between ReplicaSet and Deployment?
    Interview Answer
        A ReplicaSet is responsible for maintaining the desired number of Pod replicas, whereas a Deployment is a higher-level controller that manages ReplicaSets and provides additional features such as rolling updates, rollbacks, and version management. In production environments, Deployments are typically used instead of ReplicaSets directly.


Q) Explain Pod lifecycle?
    A Pod goes through different phases during its lifecycle. These phases indicate the current state of the Pod and its containers.

    1. Pending
        The Pod has been accepted by Kubernetes but is not yet running.
            What happens in this phase?
            - Kubernetes is finding a suitable node.
            - Pulling container images.
            - Setting up networking.
            - Mounting volumes.

            Common reasons for staying in Pending
            - Insufficient CPU or memory on nodes.
            - Image pull delays.
            - Persistent Volume not available.
            - Scheduling constraints (taints, node selectors, affinity rules).

            Troubleshooting
                # kubectl get pod
                # kubectl describe pod <pod-name>
            
            Look for:
            - FailedScheduling
            - ImagePullBackOff
            - Volume mount errors

    2. Running
        The Pod has been scheduled to a node, and its containers are running.
        
        Important:
            Running does not always mean the application is healthy.
        
        The application may still:
            - Crash internally.
            - Reject requests.
            - Fail readiness checks.
        
        To verify application health, Kubernetes uses:
            - Readiness Probes
            - Liveness Probes
            - Startup Probes

	3. Succeeded
        All containers in the Pod completed successfully and exited with status code 0.
        
        Commonly used with:
            - Jobs
            - Batch processing
            - One-time tasks
        
        Example
            A backup job finishes successfully and exits.

                NAME         READY   STATUS      AGE
                backup-job   0/1     Completed   1m

        
        Completed is the user-facing status shown by Kubernetes for the Succeeded phase.


    4. Failed
        The Pod has terminated because one or more containers exited with a non-zero status code.
        
        Common causes
            - Application crash.
            - Incorrect startup command.
            - Missing files or dependencies.
            - Permission issues.
            - Configuration errors.


    5. CrashLoopBackOff
        This is one of the most common Pod issues.
        
        What it means
            - Container starts.
            - Container crashes.
            - Kubernetes restarts it.
            - It crashes again.
            - Kubernetes waits longer before each retry (backoff).
        
        This repeated restart cycle is called a Crash Loop.

        Common causes
            - Application errors.
                - Invalid configuration.
                - Missing environment variables.
                - Syntax errors.
            - Incorrect startup command.
            - Database or external service connectivity issues.
            - Missing Secrets or ConfigMaps.
            - Port conflicts.
            - Permission or security restrictions.
            - In OpenShift, Security Context Constraints (SCC) violations are a common cause.

        Troubleshooting
                # kubectl logs <pod-name>
                # kubectl describe pod <pod-name

        Look for:
            - Look for application errors, exceptions, or configuration issues.
            - Container exit codes.
            - Error messages in logs.
            - Failed events.
            - Probe failures.
            - Volume mount issues
            - Image pull errors
            - Resource constraints (OOMKilled (Out Of Memory), CPU throttling)
            - Ensure all required ConfigMaps and Secrets exist.

    Interview One-Liner:
        A Pod lifecycle represents the different phases a Pod goes through, such as Pending, Running, Succeeded, and Failed. CrashLoopBackOff is a common error state where a container repeatedly crashes and Kubernetes keeps restarting it.


Q) How to troubleshoot CrashLoopBackOff?
    Interview One-Liner
        To troubleshoot a CrashLoopBackOff, I first check the container logs, then describe the Pod to review events and exit codes. Next, I check previous container logs, validate ConfigMaps and Secrets, verify resource limits, and ensure all external dependencies such as databases and APIs are reachable.


Q) What are the different types of Pods in Kubernetes?
    
    1. Single-Container Pod
        - Contains only one container.
        - Most common Pod type in Kubernetes.
        - Follows the "one container per responsibility" principle.
        
        Example:
        - A Pod running a single Nginx container.

    2. Multi-Container Pod
        A Pod that contains more than one container running together.
        - Containers share the same network namespace, IP address, and storage volumes.
        - Used when containers need to work closely together.
        - Common pattern: Sidecar Container
        
        Example:
        - Main application container
        - Log collector sidecar (Fluentd)
        - Monitoring sidecar

    3. Init Container Pod
        A Pod that contains one or more Init Containers.
        - Init containers run before the main application containers start.
        - They execute sequentially.
        - Used for setup and initialization tasks.
        - Must complete successfully before the application container starts.
        
        Examples:
        - Download configuration files.
        - Wait for a database to become available.
        - Create directories or initialize application data.

    4. Static Pod
        - Managed directly by the Kubelet.
        - Defined using manifest files on the node.
        - Not managed by the API Server, Deployment, or ReplicaSet.
        - Commonly used for Kubernetes control plane components.
        
        Examples:
        - kube-apiserver
        - kube-scheduler
        - etcd

        Note: Deployment, StatefulSet, DaemonSet, Job, and CronJob are not Pod types. They are controllers that create and manage Pods.

    Smallest executable unit that runs an application.      Smallest deployable unit in Kubernetes.
    Packages application code and dependencies.             Wraps one or more containers.
    Runs a single application/process.                      Provides a shared environment for containers.
    Has its own filesystem and process space.               Provides shared networking and storage to containers.

    Interview one-liner
        A container is a lightweight package that runs an application and its dependencies, whereas a Pod is the smallest deployable unit in Kubernetes that contains one or more containers and provides shared networking and storage for them. Kubernetes schedules and manages Pods, not individual containers.

Q) What is a Sidecar Container?
    - A Sidecar Container is a secondary container that runs alongside the main application container in the same Pod.
    - It extends or supports the functionality of the main container.
    - Both containers share:
        - Same network namespace
        - Same IP address
        - Shared volumes (if configured)

    Why Use a Sidecar Container?
        Instead of adding extra functionality to the application, you can offload supporting tasks to a sidecar container.
        
        Common use cases:
        - Log collection
        - Monitoring
        - Proxying traffic
        - Configuration synchronization
        - Security and service mesh features

    Interview One-Liner
        A Sidecar Container is a helper container that runs alongside the main application container in the same Pod and provides supporting functionality such as logging, monitoring, proxying, or configuration management while sharing the Pod's network and storage resources.


Q) What happens when a Pod is created?
    When a Pod is created, Kubernetes follows a series of steps to schedule and run it.
    
    1. Request Sent to API Server
        - The Pod YAML is submitted using kubectl or a controller (Deployment, Job, etc.).
        - The API Server authenticates, authorizes, and validates the request.
    
    2. Pod Information Stored in etcd
        - The Pod definition is stored in etcd.
        - The Pod is created with a Pending status.

    3. Scheduler Selects a Node
        - The Scheduler finds a suitable worker node based on:
            - CPU and memory availability
            - Node selectors
            - Taints and tolerations
            - Affinity and anti-affinity rules
    
    4. Kubelet Receives the Pod Assignment
        - The Kubelet on the selected node detects that a Pod has been assigned to it.
        - It starts the Pod creation process.

    5. Container Runtime Creates Containers
        - The container runtime (containerd/CRI-O):
            - Pulls the container image (if needed)
            - Creates containers
            - Sets up storage volumes
    
    6. Networking is Configured
        - The CNI plugin assigns an IP address to the Pod.
        - Network rules are configured.

    7. Containers Start Running
        - Application containers start.
        - Liveness, Readiness, and Startup probes begin (if configured).

    8. Pod Becomes Running
        - Once containers start successfully, the Pod status changes from Pending to Running.
        - If a readiness probe is configured and passes, the Pod is added to Service endpoints and starts receiving traffic.

    Interview One-Liner
        When a Pod is created, the API Server validates and stores it in etcd, the Scheduler assigns it to a worker node, the Kubelet instructs the container runtime to create the containers, networking is configured, and the Pod transitions to the Running state.

Q) What happens if a container crashes inside a Pod?
    When a container crashes, Kubernetes' behavior depends on the Pod's restart policy.
    1. Kubelet Detects the Container Failure
        - The Kubelet continuously monitors containers running on the node.
        - When a container exits unexpectedly, Kubelet detects the failure.

    2. Restart Policy is Checked
        The Pod's restart policy determines what happens next:
            restartPolicy: Always
        
        Possible values:
        - Always (default) → Container is restarted.
        - OnFailure → Container is restarted only if it exits with a non-zero exit code.
        - Never → Container is not restarted.

    3. Container is Restarted
        - Kubelet instructs the container runtime to restart the failed container.
        - The Pod itself is not recreated.
        - Only the failed container is restarted.

    4. Repeated Crashes → CrashLoopBackOff
        - If the container keeps crashing:
        - Kubernetes gradually increases the delay between restart attempts.
        The Pod status becomes:
            CrashLoopBackOff
    
    Interview One-Liner
        If a container crashes inside a Pod, the Kubelet detects the failure and restarts the container based on the Pod's restart policy. If the container repeatedly crashes, the Pod enters a CrashLoopBackOff state. The Pod itself is not recreated unless a controller such as a Deployment replaces it.



Q) What is a Kubernetes Deployment?
    - A Deployment is a Kubernetes controller that manages Pods and ensures the desired number of application instances are running at all times.
    - It provides declarative updates for Pods and ReplicaSets.
    - Deployments are commonly used to deploy stateless applications.

    A Deployment helps you:
    - Deploy new versions of an application.
    - Scale applications up or down.
    - Roll back to previous versions if an update fails.
    - Perform updates with minimal or no downtime.
    - Automatically replace failed Pods.

    Main Responsibilities of a Deployment
    1. Desired State Management
        - Maintains the desired number of Pod replicas.
        - Continuously compares the current state with the desired state.
        - Automatically creates or removes Pods as needed.
        example:
            replicas: 3
        If one Pod fails, the Deployment automatically creates a new Pod to maintain 3 replicas.

    2. Rolling Updates
        - Updates applications without downtime.
        - Gradually replaces old Pods with new Pods.
        - Ensures application availability during upgrades.
        - Controlled using maxSurge and maxUnavailable.
        
        Example:
            # kubectl set image deployment/myapp app=nginx:1.26

    3. Rollback
        - Restores the previous version if a deployment fails.
        - Helps recover quickly from bad releases.
        
        Command:
            # kubectl rollout undo deployment/myapp
        
        View rollout history:
            # kubectl rollout history deployment/myapp

    4. Scaling
        - Increases or decreases the number of Pods.
        - Supports both manual and automatic scaling.
        
        Manual scaling:
            # kubectl scale deployment myapp --replicas=5

        Autoscaling (HPA):
            Automatically scales Pods based on CPU, memory, or custom metrics.

    Interview One-Liner
        A Deployment is a Kubernetes controller that manages Pods and ReplicaSets, ensuring the desired number of replicas are running while providing features such as scaling, self-healing, rolling updates, and rollbacks.


Q) What is a ReplicaSet?
    - A ReplicaSet is a Kubernetes controller that ensures a specified number of Pod replicas are running at all times.
    - It continuously monitors Pods and maintains the desired number of replicas.
    - If a Pod fails or is deleted, the ReplicaSet automatically creates a new Pod.
    - If there are more Pods than desired, the ReplicaSet terminates the extra Pods.
    - ReplicaSets use labels and selectors to identify and manage Pods.

    Main Responsibilities of a ReplicaSet
    1. Maintain Desired Number of Pods
    2. Self-Healing
        - Detects failed or deleted Pods.
        - Automatically recreates Pods to maintain the desired state.
    3. Pod Selection Using Labels
        Uses label selectors to determine which Pods it manages.

    Important Note
        - In real-world environments, you usually do not create ReplicaSets directly.
        - Deployments automatically create and manage ReplicaSets.
        - When you update a Deployment, Kubernetes creates a new ReplicaSet and gradually shifts traffic to the new Pods.
    
    Deployment
    ↓
    ReplicaSet
    ↓
    Pods

    Interview One-Liner
        A ReplicaSet is a Kubernetes controller that ensures a specified number of identical Pod replicas are running at all times by automatically creating or deleting Pods as needed.

Q) What is a DaemonSet?
    - A DaemonSet is a Kubernetes controller that ensures a copy of a Pod runs on every node (or selected nodes) in the cluster.
    - When a new node joins the cluster, the DaemonSet automatically creates a Pod on that node.
    - When a node is removed, the corresponding Pod is automatically removed.
    - Commonly used for node-level services that must run on all nodes.

    Main Responsibilities of a DaemonSet
    1. Run a Pod on Every Node
        - Ensures one Pod runs on each worker node.
        - Automatically manages Pods as nodes are added or removed.

    2. Node-Level Operations
        Used for services that need access to the node itself.
        
        Examples:
        - Log collection
        - Monitoring
        - Security scanning
        - Networking plugins

    3. Automatic Node Coverage
        - New nodes automatically receive the DaemonSet Pod.
        - No manual deployment required.

    Common Use Cases
        - Monitoring
            - Prometheus Node Exporter
            - Datadog Agent
        - Networking
            - Calico
            - Flannel
        - Security
            - Falco
            - Security monitoring agents 

    Interview One-Liner
        A DaemonSet is a Kubernetes controller that ensures a copy of a Pod runs on every node in the cluster and is commonly used for logging, monitoring, security, and networking agents.


Q) What happens when you run kubectl apply?
    When you execute:
        # kubectl apply -f deployment.yaml
    
    the following steps occur:
    
    1. YAML is Sent to the API Server
        - kubectl sends the resource definition (YAML) to the Kubernetes API Server.
        - The API Server authenticates, authorizes, and validates the request.

    2. Object is Stored in etcd
        - After validation, the resource definition is stored in etcd.
        - etcd becomes the source of truth for the cluster state.

    3. Controller Detects the Desired State
        - The Deployment Controller notices that new Pods need to be created.
        - It creates a ReplicaSet, which in turn creates the required Pods.

    4. Scheduler Assigns a Node
        - The Scheduler finds a suitable worker node for each Pod based on available resources and scheduling rules.
        - The selected node is assigned to the Pod.

    5. Kubelet Creates the Pod
        - The Kubelet on the assigned node receives the Pod specification.
        - It instructs the container runtime to:
            - Pull the container image.
            - Create and start the containers.
            - Set up networking and storage.
    
    6. Pod Starts Running
        - Containers start running on the worker node.
        - Pod status changes to Running (if successful).

    7. Desired State is Maintained
        - Kubernetes continuously monitors the resource.
        - If a Pod fails or is deleted, the controller automatically creates a replacement Pod to maintain the desired state.

Interview One-Liner
    When kubectl apply is executed, the YAML is sent to the API Server, stored in etcd, controllers create the required resources, the Scheduler assigns a node, the Kubelet starts the Pod on that node, and Kubernetes continuously maintains the desired state.


Q) What is the difference between kubectl create and kubectl apply?
    1. kubectl create
        - Used to create a new Kubernetes resource.
        - Creates the resource only if it does not already exist.
        - If the resource already exists, the command fails with an error.
        - Primarily used for initial resource creation.
        
        Example:
            kubectl create -f deployment.yaml

    2. kubectl apply
        - Used to create or update Kubernetes resources.
        - Works in a declarative manner.
        - Can be executed multiple times safely.
        - Updates existing resources when changes are detected in the YAML file.
        - Maintains the desired state defined in the manifest.
        - Tracks the last applied configuration.
    
        Example:
            kubectl apply -f deployment.yaml


    Interview One-Liner
        kubectl create is used only to create new resources and fails if they already exist, whereas kubectl apply is declarative and can both create and update resources while maintaining the desired state defined in the YAML file.


Q) What are Namespaces in Kubernetes?
    - A Namespace is a logical partition within a Kubernetes cluster used to organize and isolate resources.
    - Namespaces allow multiple teams, applications, or environments to share the same cluster while keeping resources separated.
    - Resource names must be unique within a namespace, but the same name can exist in different namespaces.

    Why are Namespaces Needed?
    1. Resource Isolation
        - Resources in one namespace are logically isolated from resources in other namespaces.
        - Helps organize applications and teams within the same cluster.
        
        Example:
            - dev namespace
            - qa namespace
            - prod namespace

    2. Access Control (RBAC)
        - Permissions can be assigned at the namespace level.
        - Restricts users to only the resources they need to access.
        
        Example:
        - Developers can access only the dev namespace.
        - QA team can access only the qa namespace.
        - Production access can be restricted to administrators.

    3. Resource Quotas
        - Resource usage can be limited per namespace.
        - Helps prevent one team or application from consuming all cluster resources.
        
        You can limit:
            - CPU
            - Memory
            - Storage
            - Number of Pods
            - Number of Services

    4. Environment Separation
        Commonly used to separate environments within the same cluster.
        
        Examples:
            - Development
            - Testing
            - Staging
            - Production
        
        This helps avoid conflicts and simplifies management.

        Example Scenario
            Suppose a single Kubernetes cluster is shared by:
                - Development Team
                - QA Team
                - Production Team
            
            Without namespaces:
                - Resource names may conflict.
                - Managing permissions becomes difficult.
                - Environment separation becomes challenging.
        
        Namespaces solve these problems by providing logical isolation.

        Solution:
            With Namespace:
                Each namespace can contain its own:
                - Pods
                - Services
                - Deployments
                - ConfigMaps
                - Secrets
                - PersistentVolumeClaims
            
            independently of other namespaces.

        Common Built-in Namespaces
            default – Default namespace for user workloads.
            kube-system – Kubernetes system components.
            kube-public – Publicly readable resources.
            kube-node-lease – Node heartbeat information.


    Interview One-Liner
        A Namespace is a logical partition within a Kubernetes cluster that provides resource isolation, access control, resource quotas, and environment separation for applications and teams.



Q) Explain the Deployment Lifecycle in Kubernetes.
    or What happen internally when a deployment created?

    A Deployment goes through several stages from creation to updates and deletion.

    1. Deployment Creation
        - User creates a Deployment using a YAML file.
            # kubectl apply -f deployment.yaml
        - The Deployment object is created and stored in etcd.

    2. ReplicaSet Creation
        - The Deployment Controller creates a ReplicaSet based on the Deployment specification.

    3. Pod Creation
        - The ReplicaSet creates the required number of Pods.

    4. Scheduling and Running
        - Scheduler assigns Pods to worker nodes.
        - Kubelet starts the containers.
        - Pods move to the Running state.
    
    5. Desired State Maintenance (Self-Healing)
        - Deployment continuously ensures the desired number of replicas are running.
        - If a Pod crashes or is deleted, the ReplicaSet creates a replacement Pod.

    6. Scaling
        - The number of replicas can be increased or decreased.
            # kubectl scale deployment myapp --replicas=5
        - ReplicaSet creates or removes Pods accordingly.

    7. Rolling Update
        - When the Deployment configuration changes (e.g., new image version):
        - Deployment creates a new ReplicaSet.
        - Gradually replaces old Pods with new Pods.
        - Ensures minimal or no downtime.

    8. Rollback (If Needed)
        - If the new version has issues:
        - Deployment can revert to the previous version.

    9. Deployment Deletion
        - When the Deployment is deleted:
        - ReplicaSet is deleted.
        - Pods managed by the ReplicaSet are deleted.


Deployment Lifecycle Flow
=========================
Deployment Created
         ↓
ReplicaSet Created
         ↓
Pods Created
         ↓
Pods Scheduled & Running
         ↓
Desired State Maintained
         ↓
Scale Up / Scale Down
         ↓
Rolling Updates
         ↓
Rollback (if required)
         ↓
Deployment Deleted
    
    Interview One-Liner
        The Deployment lifecycle starts with Deployment creation, followed by ReplicaSet and Pod creation. The Deployment then maintains the desired state, supports scaling, performs rolling updates and rollbacks, and finally removes all managed resources when deleted.


Q) What are Deployment Strategies in Kubernetes?
    A Deployment Strategy defines how application updates are performed when a new version is deployed.

    1. Rolling Update (Default)
        - Gradually replaces old Pods with new Pods.
        - Ensures minimal or no downtime.
        - Controlled using:
            maxSurge
            maxUnavailable

        Example:

        Before:
        v1 v1 v1 v1

        During Update:
        v2 v1 v1 v1 
        v2 v2 v1 v1
        v2 v2 v2 v1

        After:
        v2 v2 v2 v2

        Advantages:
        - Zero/minimal downtime.
        - Safer deployments.
        - Supports rollback.

    2. Recreate
        - Terminates all existing Pods first.
        - Then creates new Pods with the updated version.

        Example:
            Before:
            v1 v1 v1 v1
                |
            Update:
            All Pods Deleted
                ↓
            v2 v2 v2 v2

        Advantages:
            - Simple strategy.
            - No version mixing.

        Disadvantages:
            - Causes downtime.

        YAML:
        strategy:
            type: Recreate

    Real-World Usage
        Rolling Update
        - Web applications
        - APIs
        - Microservices
        
        Recreate
        - Applications that cannot run multiple versions simultaneously
        - Certain database-related applications
        - Legacy applications

    Interview One-Liner
        Kubernetes supports two deployment strategies: RollingUpdate and Recreate. RollingUpdate gradually replaces old Pods with new ones with little or no downtime, while Recreate deletes all existing Pods before creating new Pods, causing downtime. RollingUpdate is the default strategy.


    Blue-Green and Canary are advanced deployment patterns implemented using:
        - Services
        - Ingress
        - Service Meshes (e.g. Istio)
        - GitOps tools like Argo Rollouts

    1. Blue-Green Deployment
        Two identical environments exist:
        
        Blue = Current production version
        Green = New version

Flow

Blue Environment (v1)
        ↓
     Users

Deploy new version:

Blue (v1)
Green (v2)

After testing Green:

Users
  ↓
Green (v2)

Traffic is switched from Blue to Green.

    Advantages
        - Near-zero downtime.
        - Quick rollback.
        - Easy testing before release.
    
    Disadvantages
        - Requires double infrastructure.
        - Higher resource cost.


    2. Canary Deployment
        - Deploy the new version to a small percentage of users first.

Example
90% Traffic → v1
10% Traffic → v2

If healthy:

70% Traffic → v1
30% Traffic → v2

Then:

100% Traffic → v2

    Advantages
        - Lower deployment risk.
        - Real user testing.
        - Easier issue detection.
    
    Disadvantages
        - More complex setup.
        - Requires traffic splitting.


    Deployment Strategy Interview Answer
        Basic Answer
        Kubernetes Deployments natively support:
        - RollingUpdate
        - Recreate
    
    Advanced Answer
        In production, organizations also use:
        - Rolling Update
        - Recreate
        - Blue-Green Deployment
        - Canary Deployment
    
    Blue-Green and Canary are not native Deployment strategies; they are deployment patterns typically implemented using Ingress Controllers, Service Meshes, or tools such as Argo Rollouts.

    Interview One-Liner
        Kubernetes supports RollingUpdate and Recreate natively. Blue-Green and Canary are advanced deployment patterns used for safer releases and are usually implemented using Services, Ingress, Service Meshes, or progressive delivery tools like Argo Rollouts.
