Q) Explain the Kubernetes Networking Model.
		
		Kubernetes networking enables communication between Pods, Services, and external clients.
		Core Principles
		1. Every Pod Gets Its Own IP Address
				- Each Pod receives a unique IP address.
				- Pods can communicate directly with other Pods using their IPs.
				- No NAT (Network Address Translation) is required between Pods.

		2. Pod-to-Pod Communication
				- Pods can communicate with each other across nodes using Pod IPs.
				- Kubernetes networking provides a flat network where all Pods can reach each other.
					Pod-A  ←→  Pod-B

		3. Pod-to-Service Communication
				- Services provide a stable IP and DNS name for accessing Pods.
				- Traffic sent to a Service is load-balanced across matching Pods.
			
				Flow:	Pod -> Service -> Pods

		4. External-to-Service Communication
				- External users can access applications through:
					- NodePort
					- LoadBalancer
					- Ingress
				
				Flow: Internet -> Ingress/LB -> Service -> Pods

		5. Service Discovery with CoreDNS
					- CoreDNS provides DNS-based service discovery.
					- Pods can communicate using Service names instead of IP addresses.
					
					Example:
								mysql.default.svc.cluster.local

				Networking Components
					Pod Network
						- Enables Pod-to-Pod communication.
						- Implemented using CNI plugins.
					
					Service Network
						- Provides stable virtual IPs for Services.
					
					DNS
						- CoreDNS resolves Service names to IP addresses.
					
					CNI Plugin
						- Responsible for network setup and Pod connectivity.
					
					Examples:
						- Calico
						- Flannel
						- Cilium

Networking Flow
Pod ↔ Pod
  ↓
Service ↔ Pod
  ↓
DNS (CoreDNS)
  ↓
Ingress / LoadBalancer
  ↓
External Users

	Interview One-Liner
		Kubernetes follows a flat networking model where every Pod gets a unique IP address and can communicate directly with other Pods. Services provide stable access to Pods, CoreDNS enables service discovery, and Ingress or LoadBalancer resources expose applications to external users.

Q) What is CNI?
		- CNI (Container Network Interface) is a standard used by Kubernetes to configure networking for containers and Pods.
		- It enables communication between Pods, Services, and external networks.
		- When a Pod is created, the CNI plugin assigns an IP address and configures network connectivity.
		- Kubernetes relies on CNI plugins to implement the cluster networking model.

		Common CNI Plugins
			- Calico
			- Flannel	
			- Cilium
			- Weave Net

		Responsibilities of a CNI Plugin
			- Assigns IP addresses to Pods.
			- Enables Pod-to-Pod communication.
			- Configures network routing.
			- Implements network policies (depending on the plugin).

		Interview One-Liner
			CNI (Container Network Interface) is a networking standard used by Kubernetes to configure Pod networking. CNI plugins assign IP addresses, enable Pod communication, and provide network connectivity within the cluster.


Q) What happens if no network policy exists?
	  	- By default, Kubernetes allows all traffic between Pods.
			- If no NetworkPolicy is defined, any Pod can communicate with any other Pod in the cluster.
			- Both ingress (incoming) and egress (outgoing) traffic are allowed.
			- This behavior is often called "default allow".

			Example
				Pod-A  ←→  Pod-B
				Pod-A  ←→  Pod-C
				Pod-B  ←→  Pod-C
			
			All Pods can communicate freely unless restricted by a NetworkPolicy.

			Interview One-Liner
				If no NetworkPolicy exists, Kubernetes allows all ingress and egress traffic between Pods by default, meaning any 
				Pod can communicate with any other Pod in the cluster.


Q) What is Ingress and Egress traffic?
			In Kubernetes networking:
				Ingress Traffic
				- Ingress means incoming traffic to a Pod or Service.
				- It is data coming into the application from outside or from other Pods.
			
				Example:
				- User sending request to a web application.
				- API calling a backend service.
						Flow: User / Pod → Pod (Application)

				Egress Traffic
				- Egress means outgoing traffic from a Pod or Service.
				- It is data going out of the application to other services or external systems.
				
				Example:
				- Application calling a database.
				- Pod accessing an external API.
					Flow: Pod (Application) → Database / External API

				Real-World Example
					Client → Web App (Ingress)
					Web App → Database (Egress)

				In Kubernetes Network Policies
					- Ingress rules control incoming traffic to Pods.
					- Egress rules control outgoing traffic from Pods.

				Interview One-Liner
					Ingress traffic is incoming traffic to a Pod or Service, while egress traffic is outgoing traffic from a Pod to 
					other services or external systems.


Q) What is Service Discovery in Kubernetes?
			- Service Discovery is the process of automatically detecting and connecting to services (applications) within a Kubernetes cluster.
			
			- It allows Pods to find and communicate with other Services without needing to know their IP addresses.

		Why Service Discovery is Needed
			- Pod IPs are dynamic (they change when Pods restart).
			- Hardcoding IP addresses is not practical.
			- Kubernetes solves this using Services + DNS.

		How Service Discovery Works
			Kubernetes provides service discovery in two ways:
			1. DNS-Based Service Discovery (Most Common)
				- Kubernetes uses CoreDNS to map Service names to IP addresses.
				- Pods can access services using DNS names.
				
				Example:
						mysql.default.svc.cluster.local
				
				or simply:
						
						mysql

			2. Environment Variables (Legacy Method)
				- Kubernetes injects Service information as environment variables into Pods.
				- Less commonly used today compared to DNS.

			Key Components Involved
				- Service → Provides stable endpoint for Pods
				- CoreDNS → Resolves service names to IPs
				- Kube-proxy → Routes traffic to backend Pods

		Interview One-Liner
			Service discovery in Kubernetes allows applications to automatically find and communicate with each other using Service names instead of IP addresses, typically implemented using Kubernetes Services and CoreDNS-based DNS resolution.


Q) What is headless service?
		Interview One-Liner
				A Headless Service in Kubernetes is a Service without a ClusterIP that returns Pod IPs directly instead of load balancing traffic, commonly used with StatefulSets and stateful applications.

Q) What is an External IP in Kubernetes?
			- An External IP is a public IP address assigned to a Kubernetes Service to allow access from outside the cluster.
			- It enables external users or systems to directly access applications running inside the cluster.
			- External IPs are usually assigned to Services of type LoadBalancer or manually configured on Services.

How It Works

Internet User
      ↓
External IP
      ↓
Kubernetes Service
      ↓
Pods
		
		When is External IP Used?
			- When you want to expose an application outside the Kubernetes cluster.
			- Common in production environments for public-facing applications.

		Important Notes
			- External IPs are usually provided by cloud providers (AWS, Azure, GCP).
			- In on-prem clusters, External IPs may need manual configuration.
			- External IP directly maps traffic to the Service, not directly to Pods.

	Interview One-Liner
			An External IP in Kubernetes is a public IP address assigned to a Service that allows external users to access applications running inside the cluster, typically provided through LoadBalancer services.

Q) How does Service Load Balancing work in Kubernetes?
		- Kubernetes Service load balancing distributes incoming traffic across multiple Pods that match the Service selector.
		- It ensures no single Pod is overloaded and improves availability and scalability.
		- Load balancing is handled mainly by kube-proxy on each node.

		How It Works Internally
		1. Service Creates a Virtual IP (ClusterIP)
			- Every Service gets a stable virtual IP called ClusterIP.
			- This IP acts as the entry point for traffic.

		2. Pod Selection using Labels
			- The Service uses labels and selectors to find matching Pods.
			- Service → selects Pods with label app=nginx

		3. kube-proxy Handles Traffic
			- kube-proxy runs on every node.
			- It watches Services and Endpoints.
			- It sets up routing rules using:
					- iptables OR
					- IPVS

		4. Traffic is Distributed
				When a request comes:

Client
  ↓
Service (ClusterIP)
  ↓
kube-proxy rules
  ↓
Pod-1 / Pod-2 / Pod-3

			Traffic is distributed using:
				- Random selection (iptables mode)
				- Efficient load-based algorithms (IPVS mode)

			Key Component
				- kube-proxy → Core component responsible for implementing Service load balancing rules.

			Important Note
				- Kubernetes does not use a single centralized load balancer inside the cluster.
				- Load balancing is distributed across all nodes via kube-proxy.

		Interview One-Liner
			Kubernetes Service load balancing works by using kube-proxy on each node, which creates routing rules (iptables or IPVS) to distribute incoming traffic across all healthy Pods selected by the Service using labels and selectors.

Q) What is TLS termination?
		Interview One-Liner
			TLS termination is the process of decrypting HTTPS traffic at the Ingress controller or load balancer before forwarding it as HTTP to backend services, simplifying certificate management and reducing load on application Pods.

Q) Explain multi-cluster networking?
		Interview One-Liner
			Multi-cluster networking enables communication between applications running in different Kubernetes clusters using networking solutions like VPN/VPC peering, service meshes such as Istio, or multi-cluster tools like Submariner and Karmada to provide global scalability, high availability, and disaster recovery.
